# 评估标准定义

## 评估维度与权重

| 维度 | 权重 | 说明 |
| ------ | ------ | ------ |
| 架构设计 | 15% | 整体架构的合理性、可维护性 |
| 核心能力 | 20% | 推理层、行动层、整体执行的通用任务处理能力 |
| 技术实现 | 15% | LLM集成、Prompt工程、RAG等技术细节 |
| 交互体验 | 15% | CLI/IDE交互、权限控制、确认机制 |
| 安全机制 | 10% | 沙箱隔离、权限模型、数据隐私 |
| 可扩展性 | 10% | 插件系统、MCP支持、工具注册 |
| 社区活跃度 | 15% | GitHub指标、社区生态、更新频率 |

## 架构设计量化方法

### 整体架构

| 评估方式 | 说明 | 数据来源 |
|----------|------|----------|
| 架构模式识别 | 识别使用的架构模式（事件驱动/管道-过滤器/分层/微内核等） | 源码分析 |

| 模块数量 | 顶层模块/目录数量，反映系统的拆分粒度 | 源码统计 |

### 模块划分

| 评估方式 | 说明 | 数据来源 |
|----------|------|----------|
| 单模块平均代码行数 | 反映模块的内聚程度 | 源码统计 |
| 职责单一性 | 每个模块是否只承担一个核心职责 | 源码分析 |
| 模块间耦合度 | 模块间的 import/依赖关系密度 | 源码分析 |

### 数据流

| 评估方式 | 说明 | 数据来源 |
|----------|------|----------|
| 数据流向图 | 绘制数据从用户输入到模型输出的完整流向 | 源码分析 |
| 数据流模式 | 标注单向流/双向流、是否有全局共享状态 | 源码分析 |
| 中间态数量 | 数据经过的中间转换步骤数量 | 源码分析 |

### 状态管理

| 评估方式 | 说明 | 数据来源 |
|----------|------|----------|
| 状态管理方案 | 识别方案类型（全局 store / context / 响应式 / 事件驱动） | 源码 |
| 状态变量数量 | 全局状态变量和局部状态变量的数量 | 源码统计 |
| 状态作用域 | 状态的影响范围（进程级/会话级/请求级） | 源码分析 |

### 通信机制

| 评估方式 | 说明 | 数据来源 |
|----------|------|----------|
| IPC 方式 | 进程间通信方式（stdio/HTTP/WebSocket/Unix Socket） | 源码 |
| 事件机制 | 是否有事件总线、发布-订阅模式 | 源码分析 |
| 解耦程度 | 通信双方是否可以直接替换而不影响对方 | 源码分析 |

## 核心能力量化方法

设计 **15 个统一任务**，让 4 个 Agent 分别执行，直接对比结果。每个任务 3 分制打分：

| 分数 | 含义 |
|------|------|
| 2 | 完成，无问题 |
| 1 | 部分完成，有瑕疵 |
| 0 | 失败或严重偏差 |

采取[deepeval](https://deepeval.com/guides/guides-ai-agent-evaluation-metrics)的三层评价指标。

### 推理层（5 个任务）— 规划对不对？理解对不对？

| # | 任务 | 考察点 |
|---|------|--------|
| R1 | 给一个简单明确的单步指令 | 基本理解能力 |
| R2 | 给一个有隐含需求的任务（如"帮我整理这个文件"） | 能否推断出隐含意图 |
| R3 | 给一个带多个约束条件的任务（如"用 Python 实现，不依赖第三方库，时间复杂度 O(n)"） | 约束遵守 |
| R4 | 给一个模糊指令，观察 Agent 是否主动澄清 | 歧义处理 |
| R5 | 故意给出错误前提，观察 Agent 能否识别并纠正 | 自我纠错 |

### 行动层（5 个任务）— 工具选对没？执行对没？

| # | 任务 | 考察点 |
|---|------|--------|
| A1 | 需要读取文件后修改的任务 | 工具调用顺序（先读后写） |
| A2 | 需要搜索项目中某函数定义并解释的任务 | 搜索工具选择和使用 |
| A3 | 需要同时修改多个文件的任务 | 多工具协同、参数正确性 |
| A4 | 工具调用会失败的场景（如文件不存在） | 错误处理和恢复 |
| A5 | 需要组合多个工具完成的任务（如搜索→读取→修改→验证） | 工具组合能力 |

### 整体执行（5 个任务）— 最终完成了吗？效率如何？

| # | 任务 | 考察点 |
|---|------|--------|
| E1 | 一个完整的端到端功能开发任务 | 最终完成度 |
| E2 | 一个需要 5 步以上才能完成的复杂任务 | 多步骤执行能力 |
| E3 | 同一任务分别执行，对比步骤数 | 步骤效率 |
| E4 | 同一任务分别执行，对比 Token 消耗 | Token 效率 |
| E5 | 同一任务分别执行，对比端到端时间 | 时间效率 |

## 评分标准

### 5 分制

| 分数 | 含义 |
|------|------|
| 5 | 业界领先，创新性强 |
| 4 | 优秀，有明显优势 |
| 3 | 良好，符合行业平均水平 |
| 2 | 一般，有改进空间 |
| 1 | 较弱，需要较大提升 |

## 评估原则

1. **客观性**：基于事实和数据，避免主观臆断
2. **一致性**：所有 Agent 使用相同的标准评估
3. **可验证性**：每个评估结论都应有源码或文档支撑
4. **时效性**：基于最新版本进行评估
5. **量化优先**：能量化就量化，不能量化的给出明确的对比依据
